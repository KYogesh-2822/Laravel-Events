ARCHITECTURE OVERVIEW
---------------------
We are using Socket.IO (Node.js) with Laravel. Since Laravel runs on PHP and
Socket.IO runs on Node.js, they cannot talk directly. Redis acts as a bridge.

Flow:
  Laravel (PHP) --publishes event--> Redis --picks up--> Socket.IO (Node.js) --pushes--> Browser

5 Layers (bottom to top):
  Layer 1: Business Logic        (existing Laravel controllers/models)
  Layer 2: Laravel Events        (broadcast events to Redis)
  Layer 3: Redis                 (message broker between PHP and Node.js)
  Layer 4: Socket.IO Server      (Node.js WebSocket server)
  Layer 5: Frontend Client       (socket.io-client in browser)

3 Separate Processes Running Together:
  - Laravel app         (port 8000)  - handles HTTP requests
  - Socket.IO server    (port 3000)  - handles WebSocket connections
  - Redis               (port 6379)  - bridges Laravel and Socket.IO


================================================================================
WHY SOCKET.IO INSTEAD OF LARAVEL REVERB?
================================================================================

We evaluated 5 options:
  1. Laravel Reverb     - Official Laravel WebSocket server (PHP)
  2. Pusher             - Hosted WebSocket service (paid)
  3. Ably               - Hosted WebSocket service (paid)
  4. Laravel Websockets - Community package (outdated, replaced by Reverb)
  5. Socket.IO          - Node.js WebSocket library (chosen)

Socket.IO was chosen because:
  - Higher performance for I/O (Node.js V8 engine)
  - Supports 50K+ concurrent connections
  - Binary data transfer support
  - HTTP long-polling fallback (works on old browsers)
  - Framework-agnostic (can be used with other projects too)
  - Massive community and ecosystem

Trade-off:
  - Requires Node.js (separate runtime from PHP)
  - More setup complexity than Reverb
  - Need Redis as a bridge between PHP and Node.js

STEP 1: INSTALL & RUN REDIS (Message Broker)                        ✓ COMPLETED
================================================================================

WHY REDIS?
  - Laravel (PHP) and Socket.IO (Node.js) are different runtimes
  - They cannot communicate directly
  - Redis Pub/Sub acts as a mailbox between them
  - Laravel publishes events to Redis, Socket.IO subscribes and picks them up
  - Speed: ~1ms delivery (much faster than HTTP calls between them)

  NOTE: Redis is NOT the only option. Alternatives:
    - HTTP API calls (Laravel → POST → Socket.IO) — slower ~50ms, blocking
    - Database polling (Laravel → DB → Node.js polls) — very slow, 1-5 sec
    - Message Queue like RabbitMQ/Kafka — complex, enterprise-grade
  Redis is best because: fastest, non-blocking, built into Laravel.

BONUS: Redis also improves caching, queues, sessions, and rate limiting.

WHAT WE DID:

  a) Installed Docker Desktop
     - Downloaded from: https://www.docker.com/products/docker-desktop/
     - Docker creates isolated containers for services like Redis
     - Why Docker? Same setup works on dev (Windows) and production (Linux server)

  b) Installed WSL2 (Windows Subsystem for Linux)
     - Command: wsl --update
     - Why? Docker containers are Linux-based. Windows needs WSL2 to run them.
     - Docker containers use Linux kernel features (namespaces, cgroups, UnionFS)
       which only exist in Linux kernel. Windows doesn't have them.
     - On Linux: Docker runs directly. On Mac: auto VM. On Windows: needs WSL2.

  c) Installed Ubuntu in WSL2
     - Command: wsl --install -d Ubuntu
     - Why? Docker needs a Linux kernel. Ubuntu provides it through WSL2.
     - Ubuntu runs as a background service, NOT a separate desktop.
     - It's like MySQL in XAMPP — runs invisibly, you never open its interface.
     - Just a terminal window if you ever need it. Docker manages it.
     - Size: ~500 MB disk, ~300 MB RAM
     - Created Linux user: (username/password you chose)

  d) Started Docker Desktop
     - Opened from Start Menu
     - Docker engine now runs using WSL2 + Ubuntu as its foundation
     - Docker Desktop must be running for containers to work

  e) Started Redis in Docker
     - Commands:
       docker pull redis:alpine
       docker run -d --name redis -p 6379:6379 --restart unless-stopped redis:alpine

     - Flag explanations:
       docker pull redis:alpine       = downloads Redis image from Docker Hub (once)
       -d                             = run in background (detached)
       --name redis                   = name the container "redis"
       -p 6379:6379                   = map PC port 6379 → container port 6379
       --restart unless-stopped       = auto-restart when Docker/PC restarts
       redis:alpine                   = smallest Redis image (~15MB)

  f) Verified Redis is running
     - docker ps                               (shows running containers)
     - docker exec -it redis redis-cli ping    (should return PONG)

USEFUL DOCKER COMMANDS:
  docker ps              - see running containers
  docker ps -a           - see all containers (including stopped)
  docker stop redis      - stop Redis
  docker start redis     - start Redis
  docker restart redis   - restart Redis
  docker logs redis      - see Redis logs
  docker rm -f redis     - delete Redis container
  docker exec -it redis sh - go inside container terminal

IMPORTANT: Docker commands are global (run from any directory).

ON PC RESTART:
  Docker Desktop auto-starts → Redis container auto-starts → Redis available
  You don't need to do anything manually.

STEP 2: CONNECT LARAVEL TO REDIS                                    ✓ COMPLETED
================================================================================

WHY THIS STEP?
  Laravel needs a PHP library to communicate with Redis.
  Also need to tell Laravel to use Redis for broadcasting events.

WHAT WE DID:

  a) Installed Predis package
     - Command: composer require predis/predis
     - Run in: project directory (C:\xampp\htdocs\project\Laravel_Api)
     - Why Predis? It's a pure PHP Redis client. No PHP extension needed.
       (The alternative "phpredis" requires a C extension installed in XAMPP)

  b) Updated .env file - two changes:

     Change 1:
       REDIS_CLIENT=predis
       (was: phpredis)
       Why: Tells Laravel to use the Predis library we just installed

     Change 2:
       BROADCAST_CONNECTION=redis
       (was: log)
       Why: Tells Laravel to send broadcast events to Redis
            "log" means events go to a log file (useless)
            "redis" means events go to Redis (where Socket.IO picks them up)

     IMPORTANT: REDIS_CLIENT and BROADCAST_CONNECTION are different settings:
       REDIS_CLIENT           = which PHP library to use (predis or phpredis)
       BROADCAST_CONNECTION   = which broadcast driver to use (redis, pusher, log)

  c) Cleared config cache
     - Command: php artisan config:clear
     - Why: Laravel caches .env values. Must clear after changes.

  d) Verified connection
     - Command: php artisan tinker
     - Then:    Illuminate\Support\Facades\Redis::ping();
     - Result:  Predis\Response\Status (means PONG - connected!)

STEP 3: LARAVEL EVENTS & BROADCASTING                               ✓ COMPLETED
================================================================================

WHY THIS STEP?
  Laravel needs to "announce" when something happens (user registers, etc.)
  These announcements are called Events. Broadcasting sends them to Redis.
  Without this step, Laravel has no way to send data to Socket.IO.

WHAT ARE EVENTS?
  An Event is a PHP class that says "something happened" and carries data.

  Example:
    Event: UserRegistered
    Data: user name, email, id

  WHY use Events?
    Without events: Controller does EVERYTHING (send email, notify admin,
    broadcast, log, etc.) — becomes huge and hard to maintain.

    With events: Controller just fires event(new UserRegistered($user))
    Then separate Listeners handle each task independently.
    Add new features by adding listeners, not editing controller.

WHAT IS BROADCASTING?
  Events normally stay inside PHP — only the server knows about them.
  Broadcasting PUSHES events outside PHP so browsers can hear them.

  Regular Event:     fires → listeners run inside PHP → done
  Broadcast Event:   fires → listeners run inside PHP
                           → ALSO sent to Redis → Socket.IO → Browser

  The ONLY difference: adding "implements ShouldBroadcast" to the event class.

WHAT ARE CHANNELS?
  Channels control WHO receives the broadcast.

  PUBLIC (Channel):
    - Anyone can listen. No authentication.
    - Use for: global notifications, public stats
    - routes/channels.php is NOT checked

  PRIVATE (PrivateChannel):
    - Only authenticated users allowed.
    - Laravel checks routes/channels.php for authorization.
    - Use for: personal notifications, user-specific data

  PRESENCE (PresenceChannel):
    - Like private, but also tracks who is connected.
    - Use for: chat rooms, online status, collaborative editing

SHOULDBROADCAST vs SHOULDBROADCASTNOW:
  ShouldBroadcast:
    - Event goes to queue (jobs table) first, then broadcasts in background
    - Controller returns response immediately (fast)
    - Needs queue worker running (php artisan queue:listen)
    - Better for production — API never depends on Redis being up
    - If Redis is down: job retries automatically, moves to failed_jobs if all retries fail
    - Retry failed jobs later: php artisan queue:retry all

  ShouldBroadcastNow:
    - Event goes DIRECTLY to Redis (no queue)
    - Controller waits for Redis confirmation (~1-3ms, unnoticeable)
    - Simpler — no queue worker needed
    - If Redis is down: controller gets error, notification is lost
    - Better for testing/learning

WHERE IS THE EVENT LISTENED?
  TWO places — simultaneously:

  Place 1: PHP Listeners (server-side, user doesn't see)
    - Created with: php artisan make:listener SendWelcomeEmail --event=UserRegistered
    - Location: app/Listeners/
    - Used for: email, DB updates, logging
    - Doesn't need Socket.IO

  Place 2: Browser Listeners (client-side, user sees)
    - Written in JavaScript: socket.on('UserRegistered', (data) => { ... })
    - Location: resources/js/ or Blade templates
    - Used for: UI updates, toast notifications, live counters
    - Needs Socket.IO

  Both run at the same time, independently.
  One fails? The other still works.

WHAT WE DID:

  a) Published broadcasting config
     - Command: php artisan config:publish broadcasting
     - Creates: config/broadcasting.php
     - What it does: Defines WHERE to send broadcasts
     - Contains connections for: Reverb, Pusher, Ably, Redis, Log, Null
     - Laravel reads this file every time a ShouldBroadcast event fires
     - We added the Redis connection manually:
         'redis' => [
             'driver' => 'redis',
             'connection' => 'default',
         ]
     - Why we didn't use "php artisan install:broadcasting":
       That command installs Laravel Reverb + Echo (we don't need them).
       We only published the config and picked what we need for Socket.IO.

  b) Created channels file
     - Command: php artisan make:channel UserChannel
     - Creates: app/Broadcasting/UserChannel.php
     - Also auto-created: routes/channels.php
     - What it does: Defines WHO can listen to private/presence channels
     - Not used yet (our event uses public Channel, not PrivateChannel)
     - Will be important when we add private channels later

  c) Created first broadcast event
     - Command: php artisan make:event UserRegistered
     - Creates: app/Events/UserRegistered.php
     - Modified it to:
       1. Added "implements ShouldBroadcast" — makes it broadcastable
       2. Added "public $user" in constructor — carries user data
       3. Changed channel to Channel('notification') — public channel

  d) Tested the event
     - Command: php artisan tinker
     - Then: event(new App\Events\UserRegistered(['name'=>'Test','email'=>'test@test.com']))
     - Result: Job created in jobs table — event queued for broadcasting
     - This confirms: Controller → Event → Queue → ready for Redis

FILES CREATED IN THIS STEP:

  config/broadcasting.php
    - Purpose: WHERE to send broadcasts (Redis config)
    - Called by: Laravel automatically, every broadcast event
    - You never call this file directly

  app/Events/UserRegistered.php
    - Purpose: WHAT happened (event with user data)
    - Called by: You, from controller — event(new UserRegistered($user))
    - ShouldBroadcast tells Laravel to send it to Redis

  routes/channels.php
    - Purpose: WHO can listen (authorization rules)
    - Called by: Laravel automatically, for private/presence channels only
    - Public channels skip this file entirely

  app/Broadcasting/UserChannel.php
    - Purpose: Class-based channel authorization logic
    - Called by: Laravel when referenced from routes/channels.php
    - Not active until we register it in channels.php

HOW ALL 4 FILES WORK TOGETHER:

  Controller calls: event(new UserRegistered($user))
       │
       ▼
  UserRegistered.php
    "I carry user data"
    "I implement ShouldBroadcast"
    "Send me on Channel('notification')"
       │
       ▼
  broadcasting.php
    "BROADCAST_CONNECTION=redis"
    "Redis connection: 127.0.0.1:6379"
       │
       ▼
  channels.php (only for private/presence channels)
    "Is this user allowed?"
    For public channels → SKIPPED
       │
       ▼
  Event data published to Redis
  Channel: "laravel_database_notification"
  Data: {"event": "UserRegistered", "user": {...}}
       │
       ▼
  Waiting for Socket.IO to pick it up (Step 4)

STEP 4: SET UP NODE.JS SOCKET.IO SERVER (next step)
================================================================================

WHY THIS STEP?
  Event data is now sitting in Redis. But nobody is reading it.
  We need a Node.js server that:
    1. Subscribes to Redis (reads events from Laravel)
    2. Accepts WebSocket connections from browsers
    3. Pushes events to the right browsers

WHAT TO DO:

  a) Create socket-server/ directory in project root
     - Command: mkdir socket-server
     - This is a separate Node.js app with its own package.json

  b) Initialize Node.js project
     - Commands:
       cd socket-server
       npm init -y
     - Creates package.json (like composer.json but for Node.js)

  c) Install required packages
     - Command: npm install socket.io ioredis dotenv
     - Packages:
       socket.io   = WebSocket server (handles browser connections)
       ioredis     = Redis client for Node.js (subscribes to Redis)
       dotenv      = Reads .env file for config (ports, Redis host)

  d) Create server.js (the actual Socket.IO server)
     - Will be created in next implementation step

  e) Start the server
     - Command: node socket-server/server.js
     - Runs on port 3000
